在Java中的abstract class, interface與Ruby中的module

這幾天去到一間公司面試實習時，
被問到了與物件導向相關的問題，
一問之下發現自己雖然說自己修過大神林軒田教授的OOP，
但是其實對於這方面的觀念還是蠻薄弱的，
也可能是因為實作經驗不夠多，或是都是自己埋首在寫code，鮮少與他人討論設計理念吧Orz

因此回來後決定重新複習物件導向觀念，
在此把所看到的東西加上自己的理解、比喻做個記錄。

物件導向程式設計

Java與Ruby這兩個語言皆是純物件導向的語言，
意思就是在這兩個語言內所有的東西都是物件！
就算是一個數字”10”在Java也是屬於Integer這個類別，而在Ruby中屬於Fixnum這個類別。
（Ruby中也有Integer這個類別，不過其再派生出Fixnum與Bignum）

而OOP的概念為：

物件導向程式設計中的每一個物件都應該能夠接受資料、處理資料並將資料傳達給其它物件，因此它們都可以被看作一 個小型的「機器」，或者說是負有責任的角色。

物件導向三大特性：封裝(Encapsulation)、 繼承(Inheritance)、 多型(Polymorphism)

而在Java與Ruby中拿來實現物件導向最常用的就是 Class
其可以擁有Member（想像成變數！）與Method（想像成Function！）
並且可以藉由這樣的一個Object來產生Instance。

Java中的abstract class與interface

在Java中另外有abstract class與interface來實現更精準的物件導向概念。
為什麼需要繼承呢？
舉個例子，要描述世紀帝國裡的「村民」與「遊俠」好了，
會發現有共通的屬性：HP、攻擊力。
那在實作這兩個Class時便會重複這兩個概念，
因此可以把「人類」這個層次提高，變成一個上層的類別，
人類這個類別就擁有HP與攻擊力。
而在實作「村民」與「遊俠」的Class時只需要繼承「人類」，
再額外加上其各自擁有的功能（村民可以建造房屋等等）

而abstract class與interface就是用來更精確描述這個現象：
在abstract class中，可以擁有自己的member，也可以有非abstarct的method，而在interface的實現中，只能夠有靜態的不能被修改的member（也就是必須是static final的，不過在interface中一般不定義member），所有的成員方法都是abstract的。從某種意義上說，interface是一種特殊形式的abstract class。

他們都無法自己產生Instance，而是需要被繼承或嵌入
在進行設計時便需要先行仔細思考，何時使用abstract class何時使用interface：

abstract class是作為一個上層的類別，例如上述例子提到的「人類」即是由abstract class來定義，
而interface則是用來定義一套完整的 功能 。
例如 建造房子 這項功能，就可以使用interface來定義，
而當我要設計某個新的角色他擁有建造房子的功能時就可以implement這個interface。

因此當我在創造「村民」這個物件時，便是要extends「人類」（abstract class）、implements「建造房子」(interface)！

小結論：abstract class就是”is a”而interface為”like a”

Ruby中的module

Ruby中的module其實就很類似於Java中的interface。
Ruby的發明人也舉過以下例子：

馬的程式物件加上了線條樣式成為斑馬物件，馬是父類別，斑馬繼承了馬類別，所以斑馬是馬的子類別，而線條樣式是另外撰寫的程式碼，若又要用同樣做法將貓加上線條使其成為老虎，貓是父類別，老虎繼承了貓類別，所以老虎是貓的子類別，而線條樣式仍然是要另外撰寫的程式碼。

為了避免重寫撰寫兩次「線條樣式」的程式碼，因此加入了module的概念
僅需要使用module撰寫一個「線條樣式」
在實作老虎與斑馬時只要mixins這個線條樣式的module即可
也讓程式碼維護時可以只專注於「線條樣式」的module之上
